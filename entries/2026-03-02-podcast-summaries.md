## 1. Problem Statement

NotebookLM generates high-quality, structured podcast summaries using a custom HBR-style prompt, but the output had no home. Summaries were being produced with no system of record, no way to cross-reference themes across shows, and no path to publishing. The goal was to build a repo that stores these summaries, enables full-text search across episodes and shows, and is structured for eventual Vercel deployment.

---

## 2. Solution

Built a Next.js 16 static site where markdown files are the database. Each show gets a folder under content/shows/. Each episode is a dated .md file with YAML frontmatter (title, date, guests, YouTube URL, top comments) and an 8-section HBR-style body generated by NotebookLM. The site statically generates a homepage, per-show episode lists, and full episode detail pages at build time — no server, no database.

Added yt-dlp to the workflow to automatically fetch top 3 YouTube comments by like count, eliminating the only manual enrichment step.

---

## 3. Architecture

content/shows/[show]/          - Markdown files (the "database")
    index.md                   - Show metadata
    YYYY-MM-DD-episode.md      - Episode with frontmatter + 8-section body

lib/markdown.ts                - getAllShows(), getShowEpisodes(), getEpisode()
                                 gray-matter + remark + sanitize-html

app/
    page.tsx                   - Homepage (show grid)
    shows/[show]/page.tsx      - Show landing page
    shows/[show]/[episode]/    - Episode detail (full 8-section render)
    not-found.tsx              - Custom 404

Fully static at build time via generateStaticParams. No runtime server required. Vercel-ready.

## 4. Component Specifications

| Component | Key Details |
|---|---|
| lib/markdown.ts | Parses frontmatter with gray-matter, converts MD to HTML with remark, sanitizes output with sanitize-html. Dates normalized to ISO YYYY-MM-DD to prevent gray-matter from auto-parsing bare YAML dates as JS Date objects. |
| Homepage | Async server component, reads all shows at build time, renders card grid with Tailwind. |
| Show page | generateStaticParams from getAllShows(), episode list sorted newest-first. |
| Episode page | Full prose render via @tailwindcss/typography. HTML sanitized in lib/markdown.ts before rendering. |
| Jest tests | 4 tests covering getAllShows, getShowEpisodes, getEpisode (content + XSS sanitization). ESM config required for remark/remark-html compatibility with ts-jest. |
| yt-dlp workflow | Fetches all comments, Python sorts by like count, top 3 inserted into frontmatter automatically. |

## 5. Lessons Learned

**NotebookLM has real limitations.** It cannot capture air dates, YouTube URLs, or comments — it only reads transcripts. The workaround: auto-enrich with web search (date, episode number, YouTube URL) + yt-dlp (top comments). The combined workflow requires zero manual steps beyond pasting the NotebookLM output.

**Tailwind v4 is a breaking change from v3.** No tailwind.config.ts. Typography plugin added via @plugin in globals.css. Worth flagging early in any Next.js scaffold.

**remark/remark-html are ESM-only.** Standard ts-jest CommonJS preset fails. Fix: ts-jest/presets/default-esm + NODE_OPTIONS in the test script.

**Bare YAML dates are a latent bug.** gray-matter parses date: 2024-01-20 (unquoted) as a JS Date object. Fix: new Date(data.date).toISOString().slice(0, 10) in the utility, and quote all dates in frontmatter.

**The tool served the purpose despite its limits.** The frustration with NotebookLM's constraints was real, but the output — a structured distillation of a podcast that reinforces memory, understanding, and personal analysis — is exactly what was needed. The build transformed a friction-prone manual process into a repeatable workflow.
